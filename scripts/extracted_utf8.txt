HITEC University Taxila  |  Advanced Database Management Systems  |  Confidential  Page 2 1.  Project Overview  
Your task is to build and publicly deploy a full -stack web application called the HITEC Smart 
University Portal, or HiSUP. The portal will handle the core operations of a university campus: student 
records, course registration, faculty management, fee proc essing, library services, and result 
management. It is modelled on the kind of system a real institution like HITEC University Taxila would 
use. 
 
The project is scoped specifically so that every topic covered in this course has a natural home inside 
the sy stem. You will start development locally using SQL Server Management Studio and then 
migrate your database to a cloud SQL Server host for live deployment. At the time of your viva the 
site must be reachable by anyone on the internet.  
 
  Project:      HITEC Smart University Portal (HiSUP)  
  Backend:      ASP.NET Core 8 with C#, using MVC or Razor Pages  
  Database:     SQL Server 2019 or later, managed through SSMS  
  Cloud DB:     Azure SQL, Railway (MSSQL plugin), or freemssqlserver.com  
  Hosting:      Any platform that gives a public HTTPS URL for an ASP.NET app  
  Marks:        100  |  Weightage: 20%  |  AI tools: allowed with documentation  
 
2.  What You Will Learn  
By the time you submit this project you should be able to do all of the following without referring to 
notes:  
 
1. Design a fully normalised relational database up to 3NF and BCNF, starting from a real -world 
problem.  
2. Write stored procedures, user -defined functions, and triggers that enforce business logic at 
the database level.  
3. Create and use views, indexes, and execution plans to improve query performance.  
4. Manage transactions correctly so that concurrent users cannot corrupt data.  
5. Implement proper database security using roles, row -level security policies, and column 
encryption.  
6. Use advanced SQL features including CTEs, window functions, dynamic SQL, PIVOT, and 
the MERGE statement.  
7. Connect a live ASP.NET Core application to SQL Server using Entity Framework Core and 
ADO.NET.  
8. Migrate a locally developed database to a cloud SQL Ser ver host and keep both environments 
in sync.  
9. Deploy a complete web application with a live public URL that stays up between submission 
and viva.  
10. Maintain a professional GitHub repository with a commit history that shows steady, 
incremental work.

HiSUP is split into seven modules. Every module contributes to a single SQL Server database named 
HiSUP_DB. The modules are designed so that each one naturally demands a different set of 
database concepts; you cannot cu t corners on one module without the corresponding concepts going 
missing from your submission.  
 
Module 1: Database Design and Schema  
Before writing any C# code, you must produce a complete database design and get it reviewed. A 
shaky schema that is fixed h alfway through the project creates far more work than taking time to get it 
right at the start.  
 
ò ER Diagram: Draw a complete ERD covering all entities, relationships, cardinalities, and both 
primary and foreign keys. Export it as a PNG file and commit it t o the docs folder. Tools such 
as dbdiagram.io or draw.io are free and work well for this.  
ò Normalisation: Document the steps to take at least three of your tables from unnormalised 
form through 1NF, 2NF, and 3NF. Show the functional dependencies at each sta ge. This 
documentation goes in the project report and will be discussed in the viva.  
ò Minimum tables required: Students, Faculty, Staff, Departments, Programs, Courses, 
Sections, Enrollments, Grades, AttendanceRecords, FeeStructure, FeePayments, 
LibraryItems, LibraryIssues, Hostels, HostelAllotments, ExamSchedule, Results, 
UserAccounts, and AuditLog. You may add mo re as needed.  
ò Constraints: Every table must have a primary key. Foreign keys must specify ON DELETE 
and ON UPDATE behaviour. Use CHECK, UNIQUE, DEFAULT, and NOT NULL constraints 
wherever they are genuinely meaningful, not just as decoration.  
 
Module 2: Sto red Procedures, Functions and Triggers  
Business logic belongs in the database, not scattered across controller methods. The objects below 
are the minimum you need to build; you should add more wherever the application needs them.  
 
ò Stored procedures, at least 15: RegisterStudent, EnrollInCourse, ProcessFeePayment, 
GenerateTranscript, CalculateSemesterGPA, MarkAttendance, AllocateHostelRoom, 
IssueLibraryBook, ReturnLibraryBook, AddExamResult, GetStudentReport, 
GetFacultyWorkload, GetD epartmentEnrollment, GenerateFeeSlip, SearchCourses.  
ò User -defined functions, at least 5: fn_CalculateCGPA, fn_GetLetterGrade, 
fn_IsLibraryItemAvailable, fn_GetOutstandingFee, fn_GetAttendancePercentage.  
ò Triggers, at least 6: trg_AfterEnrollment to update s eat count, trg_AfterGradeInsert to 
recalculate GPA, trg_AfterFeePayment to update balance, trg_AuditStudentUpdate to write 
changes to the AuditLog table, trg_PreventDuplicateEnrollment as an INSTEAD OF trigger, 
and trg_AfterLibraryReturn to restore the cop ies-available count.  
 
Every stored procedure must include proper TRY/CATCH error handling and use RAISERROR or 
THROW for meaningful error messages. Input parameters must be validated inside the procedure, 
not only in C#.

ò Full-text search: Enable full -text indexing on the LibraryItems table covering the Title and 
Author columns. The library search page must use CONTAINS or FREETEXT, not a LIKE 
with a wildc ard on both sides.  
 
Module 4: Transactions, Concurrency and ACID  
A university portal has many users acting on the same data at once. Students registering for the last 
seat in a course, two cashiers processing a fee payment, a faculty member submitting grad es while 
another reads them: all of these scenarios can go wrong if transactions are not handled properly.  
 
ò Explicit transactions: Fee payment, course enrolment, and hostel allocation must each be 
wrapped in BEGIN TRANSACTION with COMMIT and ROLLBACK insid e a TRY/CATCH 
block. The C# code that calls these procedures must also handle deadlock victim errors with a 
retry loop.  
ò Savepoints: Implement SAVE TRANSACTION with partial rollback in at least one multi -step 
workflow. Bulk result upload is a natural fit: i f one student row fails validation, roll back that 
row and continue processing the rest rather than aborting the entire upload.  
ò Isolation levels: Pick two isolation levels, such as READ COMMITTED and SERIALIZABLE, 
and write a documented test that shows wha t concurrency problem each one prevents or 
allows. Include the test script and output in the project report.  
ò Deadlock handling: Write a script that reliably produces a deadlock between two sessions. 
Log the deadlock event to the AuditLog table. In C#, catc h error 1205 and retry the operation 
up to three times before surfacing the error to the user.

session should no t give a student access to another student's grades or fee records.  
 
ò Database roles: Create four roles named db_student, db_faculty, db_admin, and db_finance. 
Grant EXECUTE on specific stored procedures only. A student role must not be able to run

the old value, the new value, the database user, and the timestamp.  
 
Module 6: Advanced SQL  
Reports and analytics pages in HiSUP must be dri ven by the database, not by loops in C# that pull 
thousands of rows and process them in memory. If you find yourself writing a foreach loop over a 
large dataset in a controller, that is a sign the logic should be in a view or a query instead.  
 
ò Common Table  Expressions: Use a recursive CTE to represent the prerequisite chain for 
courses. Use a regular CTE for at least one multi -step report query such as finding the top -
ranked student in each department.  
ò Window functions: The result report and fee analytics pages must use RANK, 
DENSE_RANK, ROW_NUMBER, NTILE, LAG, LEAD, and at least one running total built with 
SUM OVER a partition.  
ò Dynamic SQL: Build the advanced search and filter page using sp_executesql with 
parameterised  dynamic SQL. The query must be injection -safe; a raw string concatenation 
approach will lose marks.  
ò PIVOT and UNPIVOT: Generate the semester -wise attendance matrix using PIVOT so that 
each semester becomes a column. Include UNPIVOT as part of the same fea ture or in a 
separate report.  
ò MERGE: Use MERGE for bulk grade import. When a student already has a grade for a 
section, update it. When they do not, insert it. When a row is removed from the import file, 
delete the existing grade. All three actions in one statement.  
 
Module 7: Backup, Restore and Moving to the Cloud  
A database that only runs on your laptop is not finished. This module takes your project from local 
development to a live, publicly accessible state.  
 
ò Database script: The entire HiSUP_DB schema , including all tables, procedures, functions, 
triggers, views, indexes, security objects, and seed data, must be captured in a single file 
named database/HiSUP_DB_Script.sql. Any team member or examiner should be able to 
open SSMS on a fresh machine, run this file, and have a working database in under five 
minutes.  
ò Backup plan: Configure a full backup job and a differential backup job. Script both jobs so 
they can be recreated on any SQL Server instance. Include a demonstration of restoring from 
the backup  in the project report.

10.  Using AI Tools  
 
  AI tools including ChatGPT, GitHub Copilot, Claude, and Gemini are permitted.  
  Using them is not cheating. Using the m without understanding what they produced  
  and then failing to explain it in the viva is an academic problem.  
  All AI usage must be logged in docs/ai_usage_log.md and committed to GitHub.  
 
What AI Is Useful For  
ò Generating boilerplate: controller scaffolding, DbContext setup, model classes.  
ò Getting a starting structure for a stored procedure or trigger that you then adapt.  
ò Explaining SQL Server error messages and suggesting how to fix them.  
ò Understanding a concept  such as window functions or RLS policies before you write your own 
version.  
ò Reviewing code you have already written and pointing out potential problems.  
ò Generating realistic test data for the seed files.  
 
What AI Cannot Do for You  
ò Design the ERD. That requires understanding the problem domain, which is yours to reason 
through.  
ò Make the normalisation decisions. Showing functional dependencies in the report means you 
must understand them.  
ò Answer viva questions. The session is individu al, live, and without notes.  
ò Replace understanding. If you cannot walk through a piece of AI -generated code line by line in 
the viva, it will cost you marks.

CREATE OR ALTER PROCEDURE RegisterStudent

    @LastName     NVARCHAR(50),  
    @Email        NVARCHAR(100),  
    @DeptID       INT,  
    @NewStudentID INT OUTPUT  
AS

    SET NOCOUNT ON;

        BEGIN TRANSACTION;

            SET @NewStudentID = SCOPE_IDENTITY();  
        COMMIT TRANSACTION;  
    END TRY  
    BEGIN CATCH  
        IF @@TRANCOUNT > 0 ROLLBACK TRANSACTION;  
        THROW;  
    END CATCH  
END;

13.  Preparing for the Viva  
The viva is 15 marks and is conducted individually. It does not matter how large your team is; each 
person is examined separately on the full project. The session will run roughly as follows:  
 
ò The instructor opens your GitHub repository and scrolls through the commit history.  
ò The instructor opens your live site and tests core features: registering a student, enrolling in a 
course, making a payment, searching the library.  
ò SSMS is opened against y our cloud database. The instructor runs queries directly.  
ò You may be asked to write a new query, modify an existing stored procedure, or add an index 
on the spot.  
ò If the AI usage log shows significant AI involvement in a section, you will be asked to expla in it 
in detail.  
 
Questions to prepare for:  
 
35. Show me the functional dependencies for the FeePayments table and explain why it is in 3NF.  
36. What happens to a student's GPA if someone deletes a row from Grades directly without 
using your stored procedure?  
37. Open  the execution plan for this query. What does it show, and which index is responsible for 
the improvement?  
38. What isolation level does your fee payment transaction use? Why that one and not 
SERIALIZABLE?  
39. Log in as a student account and try to run SELECT * FR OM Grades in SSMS. What error do 
you get and why?  
40. Walk me through the MERGE statement in your bulk grade import, line by line.  
41. Your cloud database is unreachable right now. Show me your backup restore script and talk 
me through the steps to bring it back.  
42. This section of code was in your AI usage log. Explain what it does and what you changed 
from what the AI originally gave you.
